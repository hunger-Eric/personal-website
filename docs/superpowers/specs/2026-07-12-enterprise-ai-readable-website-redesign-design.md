# 企业客户与 AI 双重可读的个人网站重设计规格

日期：2026-07-12

状态：设计方向已确认，等待用户审阅本文件后进入实施计划

## 1. 目标

把现有“AI Native 个人档案 / 技术作品集”重构为面向中小企业老板和业务负责人的业务入口，同时保持个人身份与技术可信度。

网站必须同时服务两类读者：

1. 人类客户：能快速判断自己的流程问题是否适合 AI/自动化改造，并提交问题申请 30 分钟初步诊断。
2. AI 系统：能通过语义 HTML、稳定 URL、结构化 JSON、JSON-LD 和 `llms.txt` 准确理解身份、服务方法、案例事实、公开边界与联系方式。

核心定位：

> 把依赖人工衔接、容易出错、难以持续运行的业务流程，改造成适合企业实际情况的 AI 自动化系统。

网站不销售固定行业模板。每个客户的流程不同，服务从现状诊断开始；项目案例只用于证明方法与工程能力可以迁移到不同场景。

## 2. 已确认的用户与转化目标

主要客户：有大量人工流程、希望做 AI/自动化改造的中小企业老板或业务负责人。

典型问题信号：

- 在 Excel、文档、网站、邮件、消息平台和业务软件之间反复复制信息。
- 固定判断、整理、录入、发送或汇总工作重复发生。
- 业务资料分散，查找、交接和复核依赖个人经验。
- 现有脚本或自动化一旦失败就需要人工重来。
- 希望引入 AI，但不知道应放在哪一步，也不清楚人工审核边界。

主要转化路径：

`识别自己的流程问题 -> 理解定制改造方法 -> 查看真实项目证据 -> 提交流程问题 -> 人工筛选 -> 确认是否安排 30 分钟初步诊断`

首页主 CTA：`提交你的流程问题`。

首页辅助 CTA：`看系统如何接管流程`。

## 3. 非目标

- 不做通用自动化模板商城。
- 不把技术栈、证书、教育经历或 GitHub 活跃度作为首页价值主张。
- 不把某个项目的行业逻辑包装成唯一服务范围。
- 不虚构客户名称、Logo、推荐语、效率提升比例、ROI、线索数量或其他指标。
- 不直接公开私有 GitHub 仓库、原始设计文档、客户身份、内部路径、凭据或敏感数据。
- 不让核心项目信息只存在于动画、iframe、截图或客户端交互中。

## 4. 信息架构

### 4.1 保留与重建

| 路由 | 用途 |
| --- | --- |
| `/` | 企业客户首页与主要转化入口 |
| `/projects` | 经公开审核的案例与可迁移能力索引 |
| `/projects/[id]` | 语义完整、来源受控的案例详情 |
| `/contact` | 流程问题提交与初步诊断申请 |
| `/about` | 简短个人身份、工作方法和可信背景 |
| `/articles` | 仅保留原创且与 AI、Agent、自动化、企业流程直接相关的内容 |

`/projects` 继续使用现有稳定路径，避免无必要的 URL 迁移。页面文案使用“案例”而不是“项目档案”。

### 4.2 删除或退出公开信息架构

- 删除 `/photography` 及其详情路由、公共配置、页面组件和仅供摄影使用的资源。
- 删除 `/content`、YouTube/社交内容聚合和首页媒体区块。
- 删除 `/resume` 公共入口。
- 删除首页教育经历、证书、贡献图、泛社交入口和与客户决策无关的内容。
- 删除无实际公共用途的 `/page/[slug]` 自定义页面系统；删除前先验证没有生产内容依赖。
- `/links` 永久重定向到 `/contact`。
- 与企业定位无关的文章不进入导航、站点地图、Feed 或 AI 输出；内容审查后决定归档还是物理删除。

后台可保留，但只编辑新的身份、方法、案例和联系模型。

## 5. 首页体验与视觉系统

视觉参考：[`assets/enterprise-ai-readable-homepage-target.png`](assets/enterprise-ai-readable-homepage-target.png)。

该图只定义布局层级、交互节奏和色彩方向。图中的案例文字、流程节点和交付描述不得直接复制；实施内容必须来自第 7 节的 GitHub 事实流程。

### 5.1 色彩与组件边界

- 暖象牙纸底：`#F3EFE6` 或由主题系统映射的等价色。
- 近黑石墨：`#171916`。
- 唯一品牌强调色：焦糖琥珀 `#C47A18`。
- 边框与次级文字仅使用由纸底/石墨派生的暖灰。
- 成功、警告、错误只在真实状态表达时使用低饱和语义色，不参与装饰。
- 项目缩略图、图标、流程节点、按钮和状态使用统一的单色/琥珀体系，不保留随机蓝、绿、紫或项目私有配色。
- 复用 `components/system/*` 与 `components/motion/*`；不新增页面私有按钮、卡片或色值体系。

### 5.2 首页区块

1. **Hero**
   - 标题：`把重复、易错的人工流程，变成能持续运行的系统`。
   - 两句以内解释定制诊断、人工边界与可恢复交付。
   - 三个问题识别控件：重复录入与整理、多系统无法衔接、信息分散难以追踪。
   - 主/辅 CTA。
   - 右侧以简化流程图展示“分散输入 -> AI 处理 -> 人工审核 -> 稳定输出”。

2. **案例剧场**
   - 三个章节：现在怎么做、系统接管什么、最终交付什么。
   - 复用现有 CaseFilm/HyperFrame 能力，但服务端文字必须包含同样事实。
   - 展示人工审核、审计记录、失败恢复与交付物，不展示虚构指标。

3. **不同行业，相同的流程结构**
   - 多来源信息汇总。
   - 重复判断与录入。
   - 系统之间的数据接力。
   - 异常恢复与人工审核。

4. **真实项目证据画廊**
   - 每个案例只回答：解决的流程断点、交付的系统、证明的可迁移能力。
   - 技术栈进入详情页附录，不成为卡片主信息。
   - 首屏和画廊只展示通过公开审核的案例。

5. **工作方法**
   - 诊断现状。
   - 设计人机边界。
   - 验证真实流程。
   - 交付并持续优化。

6. **最终 CTA**
   - 进入 `/contact`，不在首页嵌入长表单。

## 6. 联系与诊断流程

`/contact` 使用渐进式表单，首屏只要求描述问题，不要求一次填写完整采购信息。

必填字段：

- 当前流程或问题描述。
- 首选联系方式（邮箱或微信二选一）。
- 隐私与联系授权确认。

可选字段：

- 公司/团队类型。
- 当前使用的工具或系统。
- 重复频率。
- 最容易出错或最耗时的步骤。
- 希望改善的结果。

提交后只承诺“收到并进行人工筛选”；不承诺自动报价、固定响应时限或必然安排会议。是否进行 30 分钟诊断由人工确认。

技术要求：服务端校验、长度限制、honeypot/速率限制、错误恢复、可访问标签、隐私说明、成功状态和失败重试。分析事件只记录 CTA、开始填写、提交成功/失败，不记录表单正文。

## 7. GitHub 项目事实链与隐私

### 7.1 事实源

每个准备公开的案例必须先读取对应仓库当前默认分支中的适用文档：

- `README.md`
- `DESIGN.md` 或领域设计文档
- `docs/PROJECT-STATE.md`
- `docs/DECISIONS.md`
- 架构、产品、运行、验收或交付文档

文档与当前仓库状态冲突时，以更具体、更新且有运行/验收证据的来源为准。

### 7.2 私有仓库安全模型

- 公共网站仓库不得提交私有仓库 URL、私有仓库名称清单、原始文档、客户身份、内部路径或抓取缓存。
- 本地只读同步使用被 `.gitignore` 排除的私有源清单和审查产物目录。
- 同步产物先生成“公开候选事实”，由人工审核后才复制到已提交的公共案例配置。
- 公共案例仅包含经确认的高层事实；私有来源标记为 `private-curated`，不提供源链接。
- 公共仓库可以提供公开源链接，但案例文案仍需经过事实审查。
- 每次案例更新在本地审查记录中保存来源路径与提交 SHA；公共输出只发布审核日期和公开状态。

### 7.3 公共案例模型

每个案例必须包含：

- `id`
- `name`
- `publicStatus`
- `sourceVisibility`: `public` 或 `private-curated`
- `purpose`
- `originalWorkflow`
- `workflowBreakpoints`
- `systemBoundary`
- `inputs`
- `processing`
- `humanReview`
- `outputs`
- `failureRecovery`
- `currentStatus`
- `transferableCapabilities`
- `evidenceArtifacts`
- `limitations`
- `reviewedAt`

没有事实依据的字段必须省略，不能由模型补齐。

### 7.4 案例选择标准

只有同时满足以下条件的项目才能进入首页：

- 与企业流程问题直接相关。
- 有足够设计/状态/验收文档支撑。
- 能证明一种与其他案例不同的可迁移能力。
- 公开内容已完成隐私审核。
- 当前状态能被准确说明，不把验证中项目写成成熟交付。

其他项目可以留在案例索引、关于页或完全不公开。GitHub 仓库数量不等于网站案例数量。

## 8. AI 可读架构

### 8.1 统一内容源

首页、案例页面、`llms.txt`、JSON 路由、JSON-LD、sitemap 和 metadata 必须从相同的已审核结构化内容模型生成。禁止为人类页面与 AI 页面维护两套不同事实。

### 8.2 公共机器路由

- `/llms.txt`：身份、服务方法、主要案例、联系与机器路由索引。
- `/.well-known/brand-facts.json`：必须成为真实可访问路由，提供规范身份、定位、服务方法、公开联系方式和更新时间。
- `/ai/services.json`：适合诊断的问题、工作方法、合作边界。
- `/ai/projects.json`：所有公开案例的精简索引。
- `/ai/projects/[id].json`：单个案例的完整公开结构化事实。

这些路由使用稳定 schema、UTF-8、明确缓存策略、canonical URL 和版本字段。`robots.ts` 允许抓取 `/ai/*` 与 `/.well-known/*`。

### 8.3 HTML 与结构化数据

- 页面关键事实由服务端渲染。
- 每页只有一个主要 `<main>` 和一个页面级 `<h1>`。
- 交互控件使用可访问名称、键盘状态和 reduced-motion 策略。
- 动画/iframe 必须有等价文本、静态 artifact 和无脚本降级。
- 根页面使用 `Person` 与 `ProfessionalService` JSON-LD。
- 服务方法使用 `Service`。
- 项目根据事实选择 `SoftwareApplication`、`CreativeWork` 或 `Article`，不能全部强制为一种类型。
- 删除摄影、旧链接页和无关项目后，同步更新 sitemap、Feed、canonical、Open Graph、JSON-LD 与 `llms.txt`。

## 9. 工程前置修复

重设计上线前必须完成：

- 升级存在高危审计结果的 Next.js 版本及可安全更新的生产依赖。
- 取消 `typescript.ignoreBuildErrors`，修复业务源码类型错误，并让生产构建真实执行类型检查。
- 清理嵌套 `<main>`、重复页面级 `<h1>` 与移动菜单滚动关闭竞态。
- 为生产环境补齐与实际部署兼容的 CSP、HSTS 和嵌入限制。
- 修复或删除当前声明但不存在的 `brand-facts.json` 路由。

## 10. 验收

### 10.1 内容与隐私

- 每个公开案例都有审核记录，且文案可追溯到允许使用的源文档。
- 公共仓库中不存在私有仓库 URL、原始私有文档、客户身份或内部路径。
- 页面中没有虚构数字、Logo、推荐语或结果。

### 10.2 人类客户体验

- 桌面与移动端首屏能在不滚动的情况下说明问题、方法和主 CTA。
- 从首页、案例详情和导航都能到达 `/contact`。
- 表单成功、失败、校验和重试路径可用。
- 无横向溢出、遮挡、不可读文字或自动关闭的关键控件。

### 10.3 AI 与搜索

- `/llms.txt`、`/.well-known/brand-facts.json`、`/ai/services.json`、`/ai/projects.json` 和每个公开项目 JSON 返回 200。
- 机器路由与人类页面事实一致。
- sitemap 不再包含已删除页面。
- 核心页面在禁用 JavaScript 后仍包含完整可索引事实。
- JSON-LD 通过结构测试，并与页面可见内容一致。

### 10.4 工程验证

- `npm run audit:architecture`
- `npm run lint`
- `npx tsc --noEmit`
- `npm test`
- `npm run build`
- 桌面与 390px 移动端浏览器 QA
- 首页 -> 案例 -> 联系表单完整交互验证
- 公共输出隐私扫描与断链检查

## 11. 实施顺序

1. 修复依赖、类型门禁和基础语义问题。
2. 建立安全的 GitHub 文档审查流程与公共案例模型。
3. 审核候选项目，确定首发案例集合与公开事实。
4. 重建身份、方法、案例和联系内容源。
5. 实现首页与 `/contact`。
6. 重构 `/projects` 与案例详情。
7. 实现 AI 机器路由和结构化数据。
8. 删除旧页面、资源和死配置，添加必要重定向。
9. 完成自动化测试、浏览器 QA、隐私审计和生产验证。
