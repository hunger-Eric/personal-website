A successful config save is not the same as a successful release.

At minimum, release feedback should include:

1. Deployment platform status (for example Vercel).
2. CI workflow status (GitHub Actions).

## Why This Matters

Without CI feedback in the admin panel, users see "Saved" but still need to open GitHub to verify whether checks actually passed.

That creates blind spots during publishing and increases operational overhead.

## Implementation Outline

We added a protected API route that reads the latest target workflow run:

1. Admin save triggers polling on `/api/admin/ci-status`.
2. API queries the latest run for the selected workflow file.
3. UI maps run states to clear feedback:
   - `queued`
   - `in_progress`
   - `completed + success`
   - `completed + failure`

Now the admin panel can show progress and outcomes without context switching.

## UX Win

The core benefit is not just another endpoint.  
It is visibility at the right point in the workflow:

1. Users know whether release checks are still running.
2. Failures are discovered immediately.
3. Save actions and release outcomes form one coherent loop.

## Next Improvements

Good follow-ups include:

1. Link each status to the exact workflow run URL.
2. Bind feedback to the commit SHA created by the same save action.
3. Show a short failure summary to shorten triage time.
